# HLDspec Terminology and Flow

**This is the canonical, authoritative system model for HLDspec.** When any other
doc disagrees with this one on terminology, ownership, or flow position, this doc
wins and the other doc should be corrected to point here.

---

## 1. System definition

HLDspec is an **agent-first workflow system**. It takes messy design/source
knowledge — especially an HLD — prepares a controlled target workspace, slices and
reviews the HLD, gates decisions through role reviewers, RunSkeptic, and human
approval, then produces a bounded execution handoff for an external build/SpecKit
agent.

HLDspec does **not** replace SpecKit.

> HLDspec prepares a target workspace and generates a **SpecKit Run Card**: a
> bounded execution handoff that tells the next agent exactly how to use SpecKit,
> which evidence to use, which checkpoints to enforce, when to run RunSkeptic, when
> to delegate to subagents, and when to stop and return to HLDspec for reassessment.

---

## 2. Ownership boundaries

| Owner | Owns |
|---|---|
| **Human Decision Owner** | Intent; risky/ambiguous decisions: architecture, source of truth, constitution, feature split/merge, implementation approval. Must not be bypassed. |
| **HLDspec** | Preparation, evidence, slicing, review, gates, planning, prompts, delegation, the Run Card. |
| **SpecKit** | Final specs, plans, tasks, and implementation artifacts. |
| **Build Agent / SpecKit Proxy** | Executes **only** from a bounded Run Card generated by HLDspec. |

Hard rules:

- HLDspec must not manually write final SpecKit specs.
- HLDspec control artifacts live in `target/.hldspec/`, never in `.specify/`.
- `target/.specify/` and `target/specs/` are SpecKit-owned.
- The source HLD is read-only evidence; only the working HLD may be modified.

## 3. Canonical terminology

### Product / workflow

- **HLDspec** — the workflow system as a whole.
- **HLDspec Product Facade** — the small public command surface (`start`, `status`,
  `review`, `continue`, `diff`, `doctor`). Hides internal scripts; prints
  decision-oriented output.
- **HLDspec Agent Session** — agent-guided session that interprets user intent and
  drives the system safely using machines and scripts as tools. Does not silently
  answer human-owned decisions.
- **Judge Agent** — conceptual owner/orchestrator of the workflow. Coordinates the
  state machine, delegates to role reviewers and subagents, stops at gates, and
  produces the next safe action. Owns state transitions, checkpoint decisions, human
  questions, subagent delegation, safety boundaries, artifact completeness, and the
  final readiness decision before handoff.
- **ProjectMachine** — the **code-level** state-machine engine the Judge Agent uses.
  Owns continue/block/stop/error decisions, machine sequencing, and state
  transitions. Does **not** own specialist judgment or human-owned decisions.
- **Scripts / Tools** — deterministic generators, validators, renderers, scanners.
  May inspect, hash, classify, generate, validate, render, prepare prompts, and run
  machine transitions. Must not silently own human decisions or become a hidden
  product state machine.
- **Target Workspace** — the one `target/` directory HLDspec creates or resumes.
- **HLDspec Control Plane** — `target/.hldspec/`: state, gates, validation, sync.
- **SpecKit Workspace** — `target/.specify/` + `target/specs/`: SpecKit-owned.
- **Execution Handoff** — the act and artifact of passing a bounded Run Card to the
  external build/SpecKit agent.
- **Reassessment Point** — the moment HLDspec is called again after a build agent
  reports back, to re-evaluate drift/state and produce the next safe action.

Important distinction:

```text
Judge Agent    = decision owner / orchestrator (conceptual)
ProjectMachine = code-level workflow engine
Scripts        = deterministic tools
```

### HLD evidence

- **Source Resources** — original external material (source HLD, design docs, repo
  references, user-provided resources). Read-only; never modified implicitly.
- **Raw HLD Evidence** — `target/targetHLD/raw/HLD.raw.md`: preserved read-only
  snapshot of the original HLD.
- **Working HLD** — `target/targetHLD/HLD.md`: normalized/conversion-ready HLD used
  by HLDspec. Cleaned/marked/converted only under safe rules; source preserved
  separately.
- **HLD Chunk** — small bounded scan unit (`raw_hld_chunks.jsonl`) used by cheap
  first-pass scanning. Not a final design boundary.
- **HLD Section** — stable logical section of the working HLD
  (`target/targetHLD/sections/`), with stable IDs/references.
- **HLD Inventory** — `target/targetHLD/inventory.{json,md}`: full list of sections
  and metadata so reviewers/agents can cite evidence without rereading everything.
- **HLD Group** — `target/targetHLD/groups.{json,md}`: a coherent work area made
  from related HLD sections. Grouped by logical coherence, source-of-truth
  boundaries, architecture boundaries, API/interface ownership, data/state
  ownership, dependency order, testability, and operations concerns — **never**
  arbitrary line-count splitting.
- **Spec Package Map** — `target/targetHLD/spec_package_map.{json,md}`: mapping from
  HLD groups to bite-size SpecKit-ready packages, showing which HLD evidence feeds
  each package.

### Review

- **Scout Agent** — cheap first-pass scanner of HLD chunks. Flags signals
  (architecture, product, governance, data/state, interface, security, operations)
  and recommends follow-up reviewers. Does not decide final boundaries. Outputs
  `raw_hld_chunks.jsonl`, `raw_hld_scan_findings.jsonl`.
- **Architecture Reviewer** — reviews components, responsibilities, boundaries,
  dependencies, interfaces, data ownership, source of truth, coupling, architecture
  risks, critical constraints. Outputs `architecture_review.{json,md}`.
- **Product Reviewer** — reviews whether planned specs are real capabilities:
  feature boundaries, user value/scenarios/stories, acceptance-readiness gaps,
  first-feature suitability, fake/context-only spec detection. Outputs
  `product_review.{json,md}`.
- **Governance Reviewer** — reviews process safety and approval readiness:
  source-HLD safety, approval gates, constitution impact, unresolved assumptions,
  human-decision requirements, paid-agent/SpecKit restrictions, conflict escalation.
  Outputs `governance_review.{json,md}`.
- **Specialist Reviewer** — optional, triggered by Scout or core reviewers:
  Interface Contract, Data/State, Security, Operations, RunSkeptic, Uncle Bob/SOLID.
- **Synthesis Agent** (Handoff Docs Generator) — summarizes role-review outputs;
  invents no new decisions; preserves blockers/conflicts. Outputs
  `role_review_summary.{json,md}` and handoff docs as needed.
- **RunSkeptic Gate** — quality gate emitting PASS/ACTION/CONFLICT (see §6).
- **SpecKit Readiness Judge** — confirms HLDspec can safely produce an execution
  handoff. Outputs `speckit_readiness_report.{json,md}`.

### Planning

- **Spec Package** — bite-size unit prepared by HLDspec for SpecKit (see §7). A
  package is a bounded unit SpecKit can turn into one useful feature/spec — not a
  heading.
- **Spec Build Plan** — the plan of planned specs/packages and their quality state.
- **Feature Dependency Graph** — legacy multi-package ordering artifact
  (`target/.hldspec/feature_dependency_graph.{json,md}`) used only when an
  explicitly approved multi-package flow requires it. Not a default start output.
- **SpecKit Invocation Queue** — legacy ordered queue
  (`target/.hldspec/speckit_invocation_queue.{json,md}`) derived from the legacy
  graph. Not a default start output.
- **SpecKit Prework Package** — the evidence/prework bundle for review/approval.
- **SpecKit Proxy Dossier** — the evidence bundle (facts/approved context/sections/
  constraints) the proxy may use.

### Execution

- **SpecKit Run Card** — the complete execution instruction contract for the next
  agent (see §8). Distinct from the dossier.
- **Build Agent / SpecKit Proxy Junior** — the external executor of a Run Card.
- **Run Card Scope / Stop Condition / Report Back** — the bounded scope, the
  conditions that force a stop, and the required report format.
- **Reassessment Trigger** — a condition that requires returning to HLDspec.

## 4. Full intended flow

```text
 1. Human intent entry        source HLD/resources, target dir, optional comment
 2. HLDspec Product Facade     start / status / review / continue / diff / doctor
 3. HLDspec Agent Session      interprets intent; invokes ProjectMachine/tools safely
 4. Judge Agent                owns orchestration, gates, delegation, next safe action
 5. Target workspace creation  creates/resumes target/
 6. Raw evidence capture       copies originals to target/targetHLD/raw/
 7. Working HLD creation       target/targetHLD/HLD.md
 8. HLD slice/dice             chunks, sections, inventory, groups, spec package map
 9. Scout Agent                scans chunks; flags which specialist reviewers needed
10. Role review pipeline       Architecture, Product, Governance, optional specialists
11. Role review summary        combines findings, blockers, conflicts, next safe action
12. SpecKit workspace init     detect and plan a real `specify/spec-kit/uvx` init
13. Spec package planning      bite-size SpecKit-ready package(s) when needed
14. Legacy graph/queue         only for explicitly approved multi-package flows
15. SpecKit prework package    evidence/prework bundle
16. RunSkeptic gate            PASS / ACTION / CONFLICT
17. Human approval gate        stops for human-owned approvals
18. SpecKit Readiness Judge    confirms readiness only if all gates pass
19. SpecKit Run Card           RUN_CARD.md + RUN_CARD.json per active package
20. Execution handoff          external build/SpecKit agent receives the run card
21. SpecKit-owned phases       constitution → specify → clarify → plan → tasks →
                               analyze → implement (only after approval)
22. Report back                files changed, questions, tests, RunSkeptic, triggers
23. HLDspec reassessment       re-evaluate; produce next safe action / next run card
```

## 5. HLD slice/dice model

Evidence refinement runs source → working → chunks → sections → inventory → groups →
spec package map:

```text
Source Resources  (read-only)
  -> Raw HLD Evidence      target/targetHLD/raw/HLD.raw.md   (read-only snapshot)
  -> Working HLD           target/targetHLD/HLD.md           (safe conversion target)
  -> HLD Chunk             raw_hld_chunks.jsonl              (cheap scan unit)
  -> HLD Section           target/targetHLD/sections/        (stable logical section)
  -> HLD Inventory         target/targetHLD/inventory.*      (cite without rereading)
  -> HLD Group             target/targetHLD/groups.*         (coherent work area)
  -> Spec Package Map      target/targetHLD/spec_package_map.* (groups -> packages)
```

HLD Groups are formed by **boundaries that matter** (source of truth, architecture,
interface/data ownership, dependency order, testability, operations) — never by line
count.

## 6. Role-review pipeline and RunSkeptic

Minimum production role path:

```text
Scout Agent
  -> Architecture Reviewer
  -> Product Reviewer
  -> Governance Reviewer
  -> Synthesis Agent / Handoff Docs Generator
  -> SpecKit Readiness Judge
```

Optional specialist reviewers (Interface Contract, Data/State, Security, Operations,
RunSkeptic, Uncle Bob/SOLID) are triggered when Scout or a core reviewer flags them.

### RunSkeptic Gate

Allowed status words — **only** these three:

- **PASS** — no blocking finding is known.
- **ACTION** — a fixable missing, stale, invalid, or incomplete artifact.
- **CONFLICT** — an unresolved design, source-of-truth, or human-owned decision.

Rules:

- Missing evidence is ACTION or CONFLICT, never PASS.
- Any unresolved ACTION/CONFLICT blocks readiness.
- RunSkeptic status must be visible in gate outputs, handoff packets, and run cards.
- RunSkeptic must use the **actual current `skeptic.md`**, not memory.

## 7. Spec package, dependency graph, and queue

A **Spec Package** is a bounded unit SpecKit can turn into one useful feature/spec.
Each package should include:

```text
package id, package name, source HLD sections, purpose, responsibilities,
dependency relationships, upstream/downstream consumers, API/interface contracts,
data/state ownership, acceptance criteria, unit/integration/e2e test expectations,
required test tools, RunSkeptic trigger conditions, evidence sources, open questions,
stop condition
```

Default start/output rule:

```text
HLDspec must plan a real SpecKit workspace init first.
.specify/ is created only by a detected SpecKit init command.
HLDspec-authored control artifacts remain under target/.hldspec/source_package/.
target/.specify/source/ is a generated mirror only.
```

Legacy ordering source of truth for explicitly approved multi-package flows:

```text
Feature Dependency Graph  (target/.hldspec/feature_dependency_graph.*)
  -> SpecKit Invocation Queue  (target/.hldspec/speckit_invocation_queue.*)
```

The queue is **derived** from the graph; there is no separately hand-built queue. If
the graph and queue disagree, classify as `CONFLICT: workflow source-of-truth
conflict`.

## 8. SpecKit Run Card / execution handoff

Between HLDspec and actual SpecKit/build execution there must be a first-class
artifact: the **SpecKit Run Card**. It is not just a prompt — it is the complete
execution handoff contract for the next agent.

HLDspec produces the Run Card only after: target workspace prepared; HLD
sliced/diced; role reviews done; package map, dependency graph, and invocation queue
exist; RunSkeptic passes or blocks resolved; human approval gate satisfied.

Difference from the dossier:

```text
SpecKit Proxy Dossier = evidence bundle  ("here are the facts/constraints")
SpecKit Run Card      = instruction contract ("execute this phase; stop here")
```

Paths:

```text
target/prompts/speckit/<package-id>/RUN_CARD.md
target/prompts/speckit/<package-id>/RUN_CARD.json
target/prompts/speckit/current_RUN_CARD.{md,json}   (optional current pointer)
```

Required Run Card JSON fields:

```json
{
  "schema_version": 1,
  "package_id": "...",
  "package_name": "...",
  "phase": "...",
  "dependency_position": "...",
  "allowed_evidence": [],
  "forbidden_actions": [],
  "required_checkpoints": [],
  "runskeptic_checkpoints": [],
  "subagents": [],
  "stop_conditions": [],
  "expected_outputs": [],
  "report_contract": [],
  "reassessment_triggers": []
}
```

Required Run Card Markdown sections:

```text
# SpecKit Run Card
## Mission
## Active Package
## Current Phase
## Dependency Position
## Approved Evidence
## Forbidden Actions
## Required SpecKit Sequence
## Question Answering Policy
## Required Subagents / Reviewers
## RunSkeptic Checkpoints
## Human Approval Boundaries
## Stop Conditions
## Expected Outputs
## Test Requirements
## Report Back Format
## Reassessment Triggers
## Next Safe Action
```

Required SpecKit sequence inside the Run Card:

```text
1. Constitution update if required and approved.
2. /speckit.specify
3. /speckit.clarify if questions exist.
4. /speckit.plan
5. /speckit.tasks
6. /speckit.analyze if required.
7. Implement only after explicit implementation approval.
```
### Slice-controlled SpecKit implementation

HLDspec does not split the HLD into partial specs to make implementation smaller.
The HLD remains the complete product source of truth.

The default SpecKit execution model is:

```text
one full HLD source package
-> one /speckit.specify
-> one /speckit.plan
-> one /speckit.tasks
-> one /speckit.analyze
-> many approved implementation slices
```

Implementation slices are execution scopes, not separate product-truth documents.

Canonical slices:

```text
FOUNDATION
WALKING_SKELETON
DOMAIN_MODEL
CONTRACTS
BUSINESS_LOGIC
PERSISTENCE
API
CLI
UI
INTEGRATION_HARDENING
```

Each implementation slice must name selected task IDs, allowed files, forbidden files,
HLD anchors in scope, deferred anchors, focused tests, prior-slice regression tests,
a report contract, and a stop condition.

No raw all-task implementation is allowed unless explicitly approved by the current
gate and human owner when required.


Question-answering policy — every SpecKit question is classified as:

- **ANSWER_FROM_EVIDENCE** — directly supported by approved HLDspec artifacts.
- **ANSWER_FROM_APPROVED_DEFAULT** — only safe, reversible defaults that do not
  affect architecture, source of truth, security, data ownership, user-visible
  scope, dependency order, or implementation approval.
- **ESCALATE_TO_HUMAN** — required for architecture boundary, source of truth,
  constitution rule, API contract, security/privacy, data ownership, user-visible
  scope, dependency order, feature split/merge, or implementation approval.

Stop conditions — the build/SpecKit agent must stop if: required evidence is missing;
package order conflicts with the dependency graph; SpecKit asks a human-owned
question; RunSkeptic returns ACTION or CONFLICT; tests fail; implementation approval
is missing; existing code contradicts the HLD/spec package; scope expands beyond
approved boundaries; it must read outside allowed evidence; or it cannot prove the
next action is safe.

Report-back format:

```text
Phase run / Files read / Files changed / Questions asked /
Questions answered from evidence / Questions answered from approved defaults /
Questions escalated / RunSkeptic status / Tests run / Failures / Scope changes /
Reassessment triggers / Next safe action / Should HLDspec reassess? yes/no
```

## 9. Reassessment loop

HLDspec does not control every keystroke after handoff. It is the
planner/gatekeeper; the build agent is the executor.

```text
HLDspec prepares and gates
  -> HLDspec generates SpecKit Run Card
  -> external build/SpecKit agent executes the run card
  -> external agent reports back
  -> HLDspec is called again to reassess (the Reassessment Point)
  -> HLDspec generates the next Run Card only if safe
```

Reassessment triggers: HLD changed; generated prompt changed; SpecKit output
changed; dependency graph changed; invocation queue stale; implementation changed
files outside approved scope; RunSkeptic ACTION/CONFLICT; a human-owned question
appears; tests fail; an architecture/product/governance assumption changes.

## 10. Deprecated / confusing terms

Do not use these as current flow wording:

```text
target-spec generation            (the post-green checkpoint is the SpecKit prework
write target specs                 approval gate, not target-spec generation)
create specs manually from HLDspec
direct scripts as product workflow (scripts are agent/maintainer tools, not the UX)
subagent without a named role
review without naming owner/type
.specify/sync as HLDspec control state (control state lives in target/.hldspec/)
```

Legacy/fallback artifacts (`target_spec_work_order.*`, `spec_branch_queue.*`) may
still exist for compatibility but are not the controlling handoff when SpecKit is
available. The controlling handoff is the SpecKit Run Card plus the invocation queue,
proxy dossier, and prework quality review.

## 12. Single-HLD source-package architecture (resolved 2026-05-27)

The "single-HLD SpecKit source-package" redesign was reviewed with RunSkeptic
(`docs/archive/REWRITE_PROMPT_RUNSKEPTIC_REVIEW_2026-05-27.md`). The Human Decision
Owner chose the **hybrid (RunSkeptic safe-rec)** direction. This section is
authoritative for that architecture and supersedes the rewrite prompt where they
differ.

### Ownership of the source package (C1/C2)

- HLDspec **authors** the approved source package under
  `target/.hldspec/source_package/` (the control plane).
- The import step **materialises a derived, read-only mirror** into
  `target/.specify/source/` for the SpecKit runner. The mirror is *generated*, never
  hand-authored, and is regenerated from `.hldspec/`; every mirrored markdown file
  carries a `GENERATED by HLDspec` banner. This preserves the §2 invariant that
  HLDspec never *authors* into `.specify/`.
- The constitution is authored here only as a **proposal**
  (`.hldspec/source_package/constitution.proposed.md`). It is applied to
  `.specify/memory/constitution.md` **only at `CONSTITUTION_APPROVAL_GATE`** — SpecKit
  remains the owner of the applied constitution.

Code: `hldspec/hld_source_package.py`, `hldspec/workspace_adapter.py`
(`source_package_dir`, `specify_source_mirror_dir`, `specify_memory_dir`).

### Model tiers (C3) — KISS two-tier with lossless projection

The operational taxonomy is **two tiers**: `MODEL_SIMPLE` (mechanical, no meaning
change) and `MODEL_SMART` (meaning can change). The four canonical tiers are retained
as a documented projection so the set can expand when a concrete need appears:

```text
MODEL_SIMPLE  <- MODEL_ROUTINE
MODEL_SMART   <- MODEL_DEFAULT, MODEL_STRONG, MODEL_CRITICAL
```

`MODEL_SIMPLE` must never approve source-truth changes, approve continuation after
meaningful SpecKit output, or resolve contradictions. Code: `hldspec/model_routing.py`.

### Spec input (C4) — single default, meaningful layering retained

A **single** SpecKit input is the default for one coherent HLD. Optional value-based
layers/steps (e.g. an infra MVP layer) are supported when a slice is large enough
*and* independently valuable, sliced by meaningful boundaries (source-of-truth,
architecture, dependency order, deployable value) — **never** by line count. The
Spec Package → Feature Dependency Graph → SpecKit Invocation Queue model (§7) and its
parity invariant are **retained** for the layered path.

### Gates

The redesign defines these gates (`hldspec/gate_validator.py`):
`HLD_SHAPING_REVIEW_GATE`, `SOURCE_PACKAGE_APPROVAL_GATE`, `CONSTITUTION_APPROVAL_GATE`,
`SPECKIT_SPECIFY_REVIEW_GATE`, `SPECKIT_PLAN_REVIEW_GATE`, `SPECKIT_TASKS_REVIEW_GATE`,
`UI_VALIDATION_GATE`, `PRE_IMPLEMENTATION_APPROVAL_GATE`, `RELEASE_OR_PUSH_GATE`. Each
gate fails closed on a missing Context Receipt, missing source refs, failed
validation, stale anchors, unsupported claims, RunSkeptic ACTION/CONFLICT (or a
missing RunSkeptic PASS where required), a Consultant BLOCK (or missing Consultant
PASS where required), and a missing human approval where required.

### Session-plane control plane

The control plane is `session_plan.json` + bounded subagent packets + machine-readable
`context_receipt.json`/`phase_report.json` + the gate validator. Roles:
`main-controller` (owns gates/continuation), `hldspec-basepack`, `target-runner`,
`consultant`. The control plane owns continuation; agents do bounded work and stop; no
packet may self-approve (`can_self_approve` is invariantly False) and every gate is
owned by the main controller. Tmux is optional UI rendered from the plan; nothing
launches without `--execute`. The public `continue` refuses (exit 3) when the gate
validator blocks; gating is opt-in on `session_plan.json` existence. Code:
`hldspec/session_control.py`, `scripts/hldspec_session_control.py`, wired in
`scripts/hldspec_agent_session.py`.

### HLD content flow

`hldspec start` generates real source-package content from the working HLD via
`build_source_package_content`: `HLD.md`, `HLD.marked.md` (stable `<!-- ANCHOR: HLD-NNN -->`
markers), `hld_reference_map.json` (anchor → heading/title/role/risk/status/lines/sha256),
and `speckit_single_spec_input.md` (one input; every requirement cites a `(HLD-NNN)`
anchor). Built on the existing `hld_map.py` parser. The single-spec input is validated
so every claim under `## Requirements` cites a known anchor; duplicate/mismatched
anchors fail the build (`anchor_integrity_errors`). Only the mirror step stamps the
`GENERATED` banner — authoritative files in `.hldspec/` never carry it. Code:
`hldspec/hld_marking.py`, `hldspec/single_spec_input.py`.

> An empty/stub HLD still builds a **well-formed** package (`ok=True`, 0 anchors) so
> `start` does not fail on a stub. Well-formed is not approval-ready — the gates
> (`SOURCE_PACKAGE_APPROVAL_GATE` etc.) are what block promotion.

### Build status

Landed and tested (`tests_v2/`): source-package shape + ownership + mirror
(`test_source_package.py`), model routing (`test_model_routing.py`), gate validator
(`test_gate_validator.py`), session-plane control plane (`test_session_control.py`),
HLD marking + single-spec input + content flow (`test_hld_marking.py`). Remaining
slices (constitution/guidelines content, target `AGENTS.md`, HLD diff/stale-check,
state-machine extension, legacy deprecation) are decomposed in the RunSkeptic review
packets and each returns to GATE.
